# План тестирования (Test Plan) для проекта "Randomizer"

## 1. Введение (Introduction)

Данный документ описывает план, стратегию и подходы к тестированию мобильного приложения "Randomizer" для платформы Android.

**Цель тестирования:** Убедиться, что приложение соответствует всем функциональным и нефункциональным требованиям, является стабильным, удобным в использовании и готовым к выпуску.

## 2. Объект тестирования (Test Items)

**Краткое описание проекта:** "Randomizer" — это мобильное приложение для Android, предназначенное для генерации различных случайных результатов.

**Основные компоненты (модули):**
*   **Генератор чисел:** Выбор случайного числа в заданном диапазоне.
*   **Выбор из списка:** Создание, сохранение и выбор случайного элемента из пользовательских списков.
*   **Бросок кубиков:** Симуляция броска одного или нескольких игральных кубиков.
*   **Подбрасывание монетки:** Симуляция подбрасывания монетки ("Орёл" или "Решка").
*   **Жребий:** Интерактивная игра по выбору "помеченных" плиток на поле.

**Атрибуты качества (согласно ISO 25010):**
*   **Функциональная пригодность:** Приложение корректно выполняет все заявленные функции.
*   **Удобство использования (Usability):** Интерфейс интуитивно понятен, пользователю легко достичь своей цели.
*   **Надежность (Reliability):** Приложение работает стабильно, без сбоев и зависаний при корректном и некорректном вводе данных.
*   **Производительность (Performance Efficiency):** Интерфейс работает плавно, анимации не вызывают задержек.
*   **Совместимость (Compatibility):** Приложение корректно отображается и работает на различных версиях Android (API 24+) и размерах экранов.

## 3. Риски (Risk Issues)

*   **Некорректная логика генерации:** Случайные числа/элементы выходят за пределы заданного диапазона или не соответствуют выбранным параметрам.
*   **Сбой при обработке пользовательского ввода:** Приложение "вылетает" при вводе некорректных данных (например, буквы вместо цифр, максимальное число меньше минимального).
*   **Проблемы с сохранением данных:** Пользовательские списки не сохраняются после закрытия приложения или сохраняются некорректно.
*   **Ошибки в UI:** Элементы интерфейса некорректно отображаются на устройствах с маленькими или очень большими экранами.

## 4. Аспекты тестирования (Features to be Tested)

**Примечание:** Этот пункт согласован с преподавателем (согласно заданию).

**Функциональные требования:**
1.  **Модуль "Число":**
    -   Возможность задать минимальное и максимальное значение диапазона.
    -   Возможность задать количество генерируемых чисел.
    -   Корректность генерации одного или нескольких чисел в указанном диапазоне.
    -   Обработка ситуации, когда минимальное значение больше максимального.
2.  **Модуль "Список":**
    -   Создание нового списка.
    -   Удаление существующего списка.
    -   Добавление элементов в список.
    -   Удаление элементов из списка.
    -   Корректность случайного выбора элемента.
    -   Корректная работа режима "Без повторений".
    -   Сохранение списков и их содержимого между сессиями.
3.  **Модуль "Кубики":**
    -   Выбор количества кубиков от 1 до 4.
    -   Ввод кастомного количества кубиков (с ограничением до 20).
    -   Корректная симуляция броска и отображение результатов.
    -   Правильный подсчет суммы выпавших значений.
4.  **Общие функции:**
    -   Копирование результата в буфер обмена.
    -   Интуитивная навигация между экранами.

**Нефункциональные требования:**
1.  **UI и Usability:**
    -   Все элементы управления доступны и понятны.
    -   Адаптивная верстка для разных размеров экрана.
    -   Плавность анимаций (бросок кубиков, подбрасывание монетки, появление плитки с результатом).
2.  **Надежность:**
    -   Приложение не должно "вылетать" при повороте экрана.
    -   Приложение не должно "вылетать" при сворачивании и разворачивании.

## 5. Подходы к тестированию (Test Approach)

Основным методом будет **ручное тестирование (Manual Testing)**. Тестировщик (напарник) будет выполнять тестовые сценарии на эмуляторе Android Studio и/или физическом устройстве.

**Виды тестирования:**
*   **Функциональное тестирование:** Проверка всех функций, описанных в п.4.
*   **Тестирование удобства использования (Usability Testing):** Оценка простоты и логичности интерфейса.
*   **Негативное тестирование:** Ввод некорректных данных для проверки механизмов обработки ошибок.
*   **Регрессионное тестирование:** Повторное выполнение ключевых тестов после внесения исправлений для гарантии того, что новые изменения не сломали старый функционал.

**Инструментальные средства:**
*   **Текстовый редактор (VS Code / Notepad++):** Для ведения документации.
*   **Android Studio:** Для сборки и запуска приложения на эмуляторе.
*   **Эмулятор Android:** Pixel 5, API 34.

*   **GitHub:** Для хранения документации и контроля версий.

## 6. Представление результатов (Pass / Fail Criteria)

Каждый тестовый сценарий (Test Case) будет оценен по следующим критериям:
*   **Pass (Пройден):** Фактический результат полностью совпадает с ожидаемым результатом.
*   **Fail (Провален):** Фактический результат не совпадает с ожидаемым, приложение аварийно завершает работу, обнаружена критическая ошибка в UI или логике.

Все результаты будут занесены в документ `Test Results.md`.

Данный план тестирования охватывает все ключевые модули и аспекты качества приложения "Randomizer". Успешное прохождение тестов в соответствии с этим планом продемонстрирует, что продукт обладает требуемыми атрибутами качества и готов к использованию.
